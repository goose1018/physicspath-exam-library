# PhysicsPath 项目 — Claude Code 工作规则

> 这是 PhysicsPath（高考物理动画题库）项目的本地 Claude Code 规则。
> 全局规则在 `~/.claude/rules/common/`。本文件覆盖项目特定的强制约束。

## 🚨 第一条铁律 — Sub-Agent 独立性规则（用户强制）

**用户原话**：「如果用户要求调用子代理 完成多层检查验证 绝对不能为了省 token 合并代理」

### 绝对禁令

- ❌ **NEVER** 合并 5 关审核（A 题目识别 + B 物理推导 + C1 视觉 + C2 视觉 + D 代码审）到 1 个 agent 调用
- ❌ **NEVER** 用 1 个"合并 agent 同时跑多关"省 token
- ❌ **NEVER** 任何 token 节省理由都不能凌驾于独立 agent 验证之上
- ❌ **NEVER** C 关只跑 1 个 agent — **2026-04-29 起强制双 C 关**

### 强制要求

- ✅ 每一关 = 一个独立的 Agent 工具调用（独立 prompt）
- ✅ **5 关审核 = 5 个独立 sub-agent**（A + B + **C1 + C2** + D，可并行启动 in single message，但 NOT 合并 prompt）
- ✅ 每完成一道题向用户**完整汇报 5 个 agent 的审核情况**（包括各自 agent ID）
- ✅ 独立性绝对优先于 token 经济
- ✅ **C1 / C2 必须用不同 prompt 视角**：C1 审"几何配置正确性"（坐标/方向/形状/原图对照），C2 审"多余 / 误导元素"（多画、错层、误标、视觉混淆）

### 为什么这个规则至关重要

实证：合并 agent 在本项目中漏掉了多个严重 bug：
- Q3（撑杆跳）：运动员从未越过横杆（合并 agent 没发现）
- Q5（正方形磁场）：case 3 圆弧画到正方形外下方（合并 agent 没发现）
- Q8（圆环转动）：旋转方向反了，"顺时针"画成"逆时针"（合并 agent 没发现）
- Q12（圆弧弹簧 4 问）：5 像素错位 + Stage 3/4 小球与 Q 视觉脱离（合并 agent 没发现）
- **2025 Q12（圆弧+板+弹簧）：圆心位置反（凹向左下 vs 右上）单 C 关漏抓**（2026-04-29）
- **2025 Q8（圆环转动）：多余 "O 轨迹圆" 误导学生**（2026-04-29 用户实测发现）

单 C 关 agent 容易"基于代码逻辑认为正确"放过视觉问题。**双 C 关用不同 prompt 视角**互补：C1 验证应有元素的正确性，C2 专门找多余/误导元素。

### 违反此规则的后果

**等同于欺骗用户**。用户已多次明确强调：
- 「你万万不可假装用了 agent 骗我」
- 「每一道题处理完 都需要向我汇报你的四个 agent 审核情况」
- 「必须调用所有 subagent 一起工作」

## 5 关审核标准（PhysicsPath 项目，2026-04-29 更新：双 C 关）

每生产或修复一道题，必须通过 **5 个独立 sub-agent** 审核（C 关拆 C1+C2 双审）：

| 关 | 名称 | 内容 |
|---|------|------|
| A | 题目识别 | paperTitle/eyebrow/title/meta/text/subQuestions/officialAnswer 是否合规 |
| B | 物理推导 | 独立推导一遍，与代码对照；公式、数值、量纲一致 |
| **C1** | **视觉直觉 #1** | drawScene 几何配置（坐标、方向、形状、对照原题图）|
| **C2** | **视觉直觉 #2** | drawScene 多余/误导元素（多画、错层、误标、视觉混淆）|
| D | 代码审 | 9 项清单：裸 < 转义 / fillRect 第一行 / KaTeX String.raw（禁 aligned）/ autoplay / image 路径 / 数值稳定 / 物理逻辑 / 时间反向 / **变量遮蔽** |

**为什么加 C2**：实证发现单 C 关 agent 容易漏抓"多余画的元素"（如 2025 Q8 圆环外的"O 轨迹辅助圆" 让学生误以为有两个圆环）。C1 专注"应有的元素是否对"，C2 专注"是否有不该出现的元素"。两个独立视角互补。

历史血泪教训（2026-04-29）：
- Q1 变量遮蔽 bug（4 关 agent 全过却漏抓 → C 关只审代数推理不够）
- Q12 KaTeX `\begin{aligned}` 不能在 displayMode:false 渲染
- 2025 Q12 圆弧圆心位置反（凹向左下 vs 右上 — 单 C 关看不清原图差异）
- 2025 Q8 多余的 "O 轨迹圆" 误导学生（单 C 关认为是物理"正确"放过）

## 🚨 drawScene 不画大段文字讲解（2026-04-30 用户强制）

**用户原话**：「不用花 token 写题目讲解 ... 你只需要确保我们花的动态图符合正确答案与物理逻辑就行了 还省时间 ... 但是下面这一栏 KaTeX 讲解必须保留」

### 绝对禁令

- ❌ **NEVER** 在 drawScene 内用 ctx.fillText 手写多行文字推导面板（"分析"清单、"4 项判断"列表、"论证 ① / ②"等）
- ❌ **NEVER** 新建 drawAnalysis / drawForceAnalysis / drawDerivation 这种"全是文字推导"的子面板函数

### 强制保留（这才是 drawScene 的本职）

- ✅ 视觉/动画元素（滑块、弹簧、轨迹、力箭头、几何小图、阶段标签）
- ✅ **答案徽章**（u≥0.92 触发的"答案 X"框）— 醒目但只一个
- ✅ 关键物理量**短标注**（紧贴几何的 "v_P=0.40"、"弹簧伸长≈0.6"、"y_max=2R" 等单行）
- ✅ phases[].formula_tex（**KaTeX 公式栏在画布下方由 q-shell 渲染** — 完整推导链放这里）
- ✅ stats / conclusions / legend / officialAnswer（这些是文字推导的归宿）

### 为什么

- 文字讲解在 KaTeX 公式栏 + officialAnswer 已覆盖；画布再写一遍是冗余
- drawScene 的核心是"动态图符合正确答案与物理逻辑"，文字讲解喧宾夺主、占用画布空间
- 节省 token，让会话能完成更多题目

### 5 关 sub-agent 审核时同步遵循

- C2 关（视觉冗余）发现"drawScene 内有大段 fillText 文字推导清单" → 标 ❌ FAIL，要求删除
- 旧题目接手时若发现，把大段文字推导子面板删掉，仅留视觉 + 关键短标注 + 答案徽章
- phases[].formula_tex 必须保留（这是 KaTeX 公式栏的源数据）

### 适用范围

2026-04-30 起所有新题目；旧题目不回炉，但接手维护时按此规则简化。

- 不省略汇报 — 每道题必须汇报**完整 5 关结果**（A/B/C1/C2/D）

## 🚨 实时进度报告纪律（2026-04-29 用户强制）

**用户原话**：「每次完成一个任务你要写进进度报告里」

每完成一个**原子动作**必须立即同步进度文档（不能累积成批量更新）：

| 动作 | 同步目标 |
|---|---|
| 完成一道题（5 关 PASS）| AUDIT_LOG 追加一行（含 5 个 agent ID + 答案 + 关键发现） |
| 修一个 bug | AUDIT_LOG 末尾"修复历史"加一段（症状 + 根因 + 修复 + 教训） |
| 删一项注册 | 同步删 exams.jsx + AUDIT_LOG 标注 |
| 改铁律 / 加新规则 | 同步 CLAUDE.md + COMMUNICATION_CONTRACT.md + ~/.claude/rules/common/ |
| 大节点（如完成一整卷） | 更新 QUEUE.md 顶部状态行 + commit 标注 |

**绝对禁止**：
- ❌ 累积 N 个动作才更新进度（之前犯过：累积 8 道题才更新 QUEUE）
- ❌ 用 todo list 替代进度文档（todo 是临时的，文档是永久的）
- ❌ commit 后才更新（commit 前所有文档必须已同步好）

**违反此规则**：等同于"对未来对话隐藏当前进度"，会造成下一次接手时信息不全。

## 🚨 第二条铁律 — Sub-Agent 测试套件完整性规则（用户强制）

**用户原话**：「让 subagent 运行完整的测试套件，并报告所有失败项 必须让子代理运行完所有测试 must 必须」

### 绝对要求

- ✅ **MUST** 让 sub-agent 运行**完整**测试套件
- ✅ **MUST** sub-agent 报告**所有**失败项（不只前几个、不仅关键项、不截断）
- ✅ **MUST** sub-agent 必须**跑完所有测试**才能返回（不允许中途退出）
- ✅ **MUST** 输出明确：总数/通过数/失败数 + 每个失败项详情

### 绝对禁令

- ❌ **NEVER** 让 sub-agent 跑部分测试就报"完成"
- ❌ **NEVER** 因测试时间长跳过 / 缩减
- ❌ **NEVER** 第一个测试失败就停（必须继续到底）
- ❌ **NEVER** 选择性报告失败项

### 调用 sub-agent 跑测试的 prompt 必须包含

```
任务：运行完整测试套件
要求：
1. 运行所有测试（具体命令）
2. 跑完所有测试 — 即使有失败也不要中途退出
3. 报告所有失败项 — 不截断、不省略
4. 输出格式：总数 / 通过 / 失败 + 每个失败的测试名 + 错误信息 + 文件路径
```

### 为什么这个规则至关重要

- 测试是真理的最后一道防线
- 部分测试报告会让你以为代码 OK，但隐藏的失败可能在生产中爆炸
- 用户已多次警告"不可假装"，sub-agent 不跑完所有测试就是另一种形式的假装

**违反此规则等同于欺骗用户**。在所有项目中遵守。
